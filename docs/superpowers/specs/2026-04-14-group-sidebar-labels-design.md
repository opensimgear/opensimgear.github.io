# Group Sidebar Labels Design

## Goal

Allow autogenerated Starlight sidebar groups to display a human-authored label from each section landing page instead of
always using the directory name.

This change targets the docs tree where sections like `components/`, `guides/`, and `diy/` are autogenerated from
subdirectories under `src/content/docs/docs/`.

## Current Context

`astro.config.mjs` currently configures docs navigation with:

- a top-level `Docs` sidebar group
- `autogenerate: { directory: 'docs' }`

Within that autogenerated tree, nested group labels currently come from directory names. That means folder names become
reader-facing labels even when the content wants a more editorial label.

Example:

- `src/content/docs/docs/diy/index.md` currently uses `title: DIY Reference`
- the generated group label still reflects the directory name `diy`

This produces a mismatch between the landing page intent and the sidebar navigation.

## Constraints

- Keep the existing Starlight autogeneration model for section contents.
- Do not require a separate config-side label map.
- Make labels content-owned so editors can change them near the section landing page.
- Preserve current behavior as fallback when no custom label is provided.
- Do not change sidebar ordering behavior already verified in `src/tests/docs/sidebar-order.test.ts`.

## Options Considered

### Option A - Recommended: Use `sidebar.label` in each section `index.md`

Read the frontmatter of each autogenerated group landing page and use `sidebar.label` when present.

Pros:

- Keeps navigation label next to the section content it describes
- Decouples page title from sidebar grouping label
- Avoids duplicating section names in code config
- Gives a clean fallback path when metadata is absent

Cons:

- Requires custom logic around the autogenerated sidebar tree
- Introduces one new piece of optional frontmatter to document

### Option B - Use landing page `title`

Use the `title` field from each section `index.md` as the group label.

Pros:

- No new metadata field needed
- Easy for editors to discover

Cons:

- Couples page heading and sidebar label forever
- Removes flexibility when page title should differ from compact nav label

### Option C - Maintain a label map in code

Keep autogeneration as-is but override labels through a manual map in config or helper code.

Pros:

- Centralized override logic
- No content schema changes

Cons:

- Duplicates directory structure in code
- Easy for content and navigation to drift apart
- Less friendly for content-only edits

## Chosen Approach

Use Option A.

Each first-level autogenerated group under `src/content/docs/docs/` should be able to declare a `sidebar.label` in its
`index.md`. The sidebar generation layer should prefer that label, then fall back to the current directory-derived label
when the field is missing.

## Proposed Behavior

For each autogenerated docs subgroup:

1. Identify its landing page, typically `<group>/index.md`
2. Read frontmatter metadata from that file
3. If `sidebar.label` exists and is non-empty, use it for the sidebar group label
4. Otherwise keep the existing folder-name label behavior

Example intended metadata:

```md
---
title: DIY Reference
sidebar:
  label: DIY Reference
  order: 4
---
```

This lets the group label become `DIY Reference` without forcing the folder name to change.

## Implementation Shape

The implementation should stay narrow:

- add a helper that can resolve a custom label for an autogenerated subgroup from its `index.md`
- apply that helper only where the docs autogeneration tree is assembled or transformed
- avoid rewriting the full sidebar structure by hand if a focused post-processing step is sufficient

If Starlight exposes the generated sidebar structure at build time, the preferred implementation is to transform the
generated items after autogeneration and replace only matching group labels. If that is not possible, the fallback is a
small custom sidebar builder for the `docs` section that preserves the existing output shape.

## Data and Error Handling

- Missing `index.md`: keep existing directory-name label
- Missing `sidebar.label`: keep existing directory-name label
- Blank or whitespace-only `sidebar.label`: treat as missing
- Frontmatter parse failure: fail loudly during build rather than silently producing partial navigation

This keeps ordinary omissions safe while still surfacing broken content metadata.

## Testing Strategy

Follow TDD for the behavior change.

Add focused tests that cover:

- custom label present -> generated group uses `sidebar.label`
- custom label absent -> generated group falls back to directory name
- existing `sidebar.order` expectations remain unchanged

Tests should exercise the real helper or sidebar transformation logic, not only string parsing in isolation.

## Content Updates

Update section landing pages that want a human-friendly group label. Initial likely targets:

- `src/content/docs/docs/components/index.md`
- `src/content/docs/docs/guides/index.md`
- `src/content/docs/docs/diy/index.md`

Only add `sidebar.label` where a custom label is desired. Sections that are already acceptable can rely on fallback.

## Out of Scope

This design does not include:

- renaming directories
- changing top-level sidebar groups in `astro.config.mjs`
- changing per-page sidebar order rules
- broader navigation redesign

## Risks and Mitigations

### Risk: Starlight does not expose an easy sidebar transform point

Mitigation:

- start with a failing test around current generated output
- choose the smallest integration point that can rewrite labels without replacing unrelated sidebar behavior

### Risk: Content authors forget to add `sidebar.label`

Mitigation:

- fallback to existing directory-name behavior
- document the field by example in updated landing pages

### Risk: Label metadata drifts from landing page title

Mitigation:

- allow intentional divergence, since that is the feature
- keep initial usage small and obvious so editorial differences stay deliberate

## Implementation Plan Boundary

The later implementation plan should cover:

1. locating the Starlight integration point for autogenerated docs group labels
2. adding failing tests for custom-label and fallback behavior
3. implementing the smallest sidebar label override that satisfies those tests
4. updating section landing pages with `sidebar.label` where needed
5. verifying the site with `pnpm build`
