# Breadcrumb SEO Page Titles Design

## Goal

Improve SEO and browser-tab clarity by generating richer HTML page titles from page context while keeping the visible
page UI titles unchanged.

Example target:

- `/gear/` should render HTML title `Gear Overview | OpenSimGear`
- visible page title can remain `Overview`

## Current Context

This site uses Starlight. `src/components/overrides/Head.astro` currently delegates directly to the default Starlight
head component.

Starlight currently builds the document title as:

- `<page title> <delimiter> <site title>`

For section landing pages with generic titles like `Overview`, this produces weak titles such as:

- `Overview | OpenSimGear`

That title is acceptable for UI labeling, but it is not ideal for SEO because the most meaningful keyword context
appears too late or not at all.

## SEO Requirements

- Keep the visible page heading and sidebar labels unchanged unless explicitly edited in content
- Improve only metadata generation by default
- Prefer keyword-bearing context near the front of the HTML title
- Keep titles readable in browser tabs and search results
- Keep titles unique across the site
- Preserve Starlight defaults as fallback when breadcrumb context cannot be resolved

## Constraints

- Do not require authors to add new frontmatter for ordinary pages
- Reuse existing Starlight route and sidebar data where possible
- Keep implementation narrow to title and related metadata generation
- Avoid changing canonical URL behavior, description behavior, or visible page structure
- Support both custom-built sidebar sections and Starlight autogenerated sections

## Options Considered

### Option A - Recommended: Context-aware SEO titles from current page plus breadcrumb labels

Resolve the current page in the sidebar tree, collect parent labels, and generate an SEO title using a small
normalization rule for generic page names.

Examples:

- `Overview` under `Gear` -> `Gear Overview | OpenSimGear`
- `SilentChill oDrive Belt Tensioner` under `3rd Party` -> `SilentChill oDrive Belt Tensioner | 3rd Party | OpenSimGear`

Pros:

- Uses existing navigation labels as source of truth
- Improves generic titles without changing UI copy
- Works for both sidebar-authored and autogenerated labels
- Preserves editor flexibility without new metadata

Cons:

- Needs a small helper to locate the current page and derive parent labels
- Needs a careful rule set for generic page names to avoid awkward phrasing

### Option B - Pure breadcrumb order in the title

Generate titles directly as:

- `<page title> | <parent label> | <site title>`

Example:

- `Overview | Gear | OpenSimGear`

Pros:

- Very simple implementation
- Mirrors navigation hierarchy exactly

Cons:

- Not best SEO for generic titles because the strongest keyword may appear later
- Can produce less natural titles for section indexes

### Option C - Content-owned SEO title override field

Add a frontmatter field for explicit metadata title control on each page.

Pros:

- Maximum editorial control
- Easy to tune high-value pages individually

Cons:

- High maintenance burden
- Easy for UI title and metadata title to drift unintentionally
- Unnecessary for the current problem scope

## Chosen Approach

Use Option A.

The site should generate metadata titles from the current page title, its breadcrumb context, and the site title. For
pages with generic labels such as `Overview`, the generator should rewrite the metadata title into a more
keyword-forward phrase using the nearest meaningful parent label.

## Proposed Title Rules

### Rule 1: Specific page titles stay page-first

If the page title is already descriptive, keep it first.

Examples:

- `Actuator Motor Sizing | Calculators | OpenSimGear`
- `Privacy Policy | OpenSimGear`

### Rule 2: Generic titles become contextualized phrases

If the page title is a generic label like `Overview`, rewrite it using the nearest parent label.

Examples:

- `Overview` under `Gear` -> `Gear Overview | OpenSimGear`
- `Overview` under `3rd Party` -> `3rd Party Overview | OpenSimGear`

This keeps the UI label short while giving search engines a clearer topical phrase.

### Rule 3: Brand stays last

Use `|` as the default separator and keep `OpenSimGear` last.

Rationale:

- `|` is the clearest and most standard separator for titles
- trailing brand keeps topical keywords earlier in the string
- this pattern is familiar in tabs and search snippets

### Rule 4: Avoid redundant repetition

Do not emit duplicated adjacent concepts.

Examples to avoid:

- `Overview | Gear | OpenSimGear`
- `Gear | Gear | OpenSimGear`

Examples to emit instead:

- `Gear Overview | OpenSimGear`
- `Gear Setup Guide | OpenSimGear` if later rule expansion adds similar contextual rewrites

## Metadata Scope

### HTML `<title>`

Apply the new title generation here.

### Open Graph and Twitter title

Use the same SEO-aware computed title by default so link previews match the page title strategy.

### Meta description

Keep the current behavior unchanged. Descriptions should remain content-authored and not be built from breadcrumbs.

### Visible H1

Keep the current visible page title unchanged for now. This preserves the existing editorial and UI structure while
improving metadata independently.

## Implementation Shape

The implementation should remain focused:

- add a helper that inspects `Astro.locals.starlightRoute`
- locate the current page within the sidebar tree
- extract parent labels for breadcrumb context
- normalize generic titles like `Overview` into parent-led phrases for metadata
- replace only title-related tags in the custom head override
- leave other head tags on the current Starlight path as much as possible

The preferred implementation is a small utility such as `src/utils/page-title.ts` plus a custom
`src/components/overrides/Head.astro` that renders the computed title in place of Starlight's default `<title>`,
`og:title`, and matching social title tags.

## Data and Error Handling

- page found in sidebar and title is specific -> use page title plus parent context as needed
- page found in sidebar and title is generic -> rewrite using parent label
- page not found in sidebar -> fall back to Starlight's existing title behavior
- parent label missing -> fall back to Starlight's existing title behavior
- duplicate labels after normalization -> deduplicate before joining

This keeps the change safe even for pages outside ordinary sidebar flows.

## Separator Guidance

Default separator:

- `|`

Alternative acceptable separators:

- `-` if a lighter visual tone is wanted later
- `·` only for brand/editorial preference, not recommended as primary default for SEO-focused metadata

## Testing Strategy

Follow TDD.

Add focused tests for the title builder that cover:

- generic section index page -> `Gear Overview | OpenSimGear`
- descriptive child page -> `Page Title | Parent Label | OpenSimGear`
- custom-labeled section such as `3rd Party` -> label preserved in title
- fallback when page cannot be found in sidebar -> existing Starlight pattern
- deduplication when normalized title would repeat parent label

If feasible, test the helper in isolation and add at least one integration-level assertion that the rendered head
contains the computed `<title>`.

## Out of Scope

This design does not include:

- changing visible page titles or H1s
- bulk rewriting frontmatter titles
- changing meta descriptions
- changing canonical URLs
- implementing per-page manual SEO title overrides
- broader content strategy or keyword research work

## Risks and Mitigations

### Risk: Sidebar traversal misses some page types

Mitigation:

- keep fallback to existing Starlight title generation
- cover custom and autogenerated sidebar structures in tests

### Risk: Generic-title rewrite rules become too broad

Mitigation:

- start with a narrow list led by `Overview`
- keep normalization helper small and explicit
- avoid language generation or heuristic-heavy rewriting

### Risk: Social preview titles diverge from future editorial preferences

Mitigation:

- keep logic centralized in one title helper
- allow future extension to separate HTML and OG title behavior if needed

## Implementation Plan Boundary

The later implementation plan should cover:

1. adding a failing test for generic breadcrumb-aware SEO titles
2. building a helper that derives parent context from the current sidebar tree
3. implementing explicit normalization for generic titles such as `Overview`
4. updating the head override to emit computed `<title>`, `og:title`, and social title tags
5. preserving fallback behavior for pages without breadcrumb context
6. verifying with `pnpm build`
